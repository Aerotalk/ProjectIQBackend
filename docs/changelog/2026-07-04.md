# Changelog: 2026-07-04 (Phase 2 - Enterprise Tenant Architecture)

## 🚀 What's New

### 1. The Organization (Root Tenant)
- **New Entity:** Created `Organization` to act as the absolute root of the hierarchy.
- **Role:** Represents the overarching corporate entity that owns everything else.

### 2. Company Refactor (Child of Organization)
- **Strict Relationship:** Refactored `Company` to belong to an `Organization` (`@ManyToOne`).
- **Normalized Schema:** Removed authentication and address fields from Company, isolating it to pure legal and tax information (GST, PAN, TAN, CIN, MSME, IEC).

### 3. Addresses and Banking
- **CompanyAddress:** A new entity attached to a `Company` supporting multiple locations (Registered vs. Corporate).
- **CompanyBankAccount:** A new entity attached to a `Company` storing highly specific financial details (IFSC, SWIFT, UPI).
- **Atomic API:** Refactored the `/api/admin/companies` endpoint to accept arrays of addresses and bank accounts in a single request, creating them all in a single transaction.

### 4. Company Management (Full CRUD & Validation)
- **Validation Constraints:** Integrated `spring-boot-starter-validation` and applied `@Valid`, `@NotBlank`, and `@Email` constraints to ensure bad data is rejected before saving.
- **Complete Lifecycle APIs:** Built out the remaining REST endpoints to fully manage companies:
  - `GET /api/admin/companies`: Fetch all companies scoped to the logged-in user's Organization.
  - `GET /api/admin/companies/{id}`: Fetch detailed company data (including addresses/banks).
  - `PUT /api/admin/companies/{id}`: Update Branding (logos, colors), Tax Information (GST, PAN, etc.), Financial Information, and Contact Information dynamically.
  - `PATCH /api/admin/companies/{id}/status`: Toggle company active/inactive status quickly.
  - `DELETE /api/admin/companies/{id}`: Trigger soft-deletion for a company.

### 5. Employee Management Foundation
- **Department:** Created a `Department` entity (supporting parent-child sub-departments). Built out `DepartmentService` and `DepartmentController` for full CRUD lifecycle and nested hierarchy support.
- **Designation:** Created a `Designation` entity with built-in hierarchy levels. Built out `DesignationService` and `DesignationController` for complete CRUD management.
- **Employee:** Created the core `Employee` profile entity, linking it to an `Organization`, `User` (Auth), `Department`, and `Designation`. Also supports self-referencing to track reporting managers.
- **Employee Service:** Built out `EmployeeService` and `EmployeeController` featuring strict DTO validation, status toggling, soft deletion, and a custom `@Query` for dynamic search and filtering by name, code, department, and status.
- **Employee ID Generator:** Built an auto-generation algorithm (`generateEmployeeCode`) that automatically generates sequential employee codes based on organization headcount.

### 6. Dashboard Analytics Service
- **Aggregation API:** Built `DashboardService` and `DashboardController` (`GET /api/admin/dashboard/metrics`) to aggregate system-wide metrics (Total Companies, Employees, Departments, Roles) using fast database-level `COUNT()` queries.
- **API Mocking:** Stubbed out responses for `Applications` and `Activity Logs` to prevent frontend breakage while those modules pend implementation.

### 7. Role Management Service
- **Role CRUD:** Built `RoleService` and `RoleController` to manage the core `Role` entities.
- **Clone Action:** Implemented an endpoint to cleanly clone an existing Role and save it as a new distinct entry.
- **Assign Action (Surrogate Key):** Replaced the implicit JPA `@ManyToMany` mapping with a dedicated `UserRole` bridge entity to explicitly define a surrogate UUID primary key (`user_role_id`), perfectly matching the `user_roles` schema. Updated `RoleService` and `AdminService` to construct and insert these specific bridge entities when assigning roles.

### 8. Permission Management Service
- **Flattened Schema:** Deleted the over-engineered nested `Module` entity structure. Refactored the `Permission` entity to perfectly match the flat `permissions` database schema (using `permission_id`, `permission_key`, `permission_name`, `module`, and `description`).
- **Role Permissions (Surrogate Key):** Replaced the implicit JPA `@ManyToMany` mapping with a dedicated `RolePermission` bridge entity to explicitly define a surrogate UUID primary key (`role_permission_id`), perfectly matching the `role_permissions` schema. Refactored `PermissionService` to construct and insert these specific bridge entities instead of direct set assignments.
- **Dynamic Grouping API:** Built `PermissionService` and `PermissionController` exposing `/api/admin/permissions/matrix`. Instead of eager-fetching a deep JPA hierarchy, it now executes a highly performant `SELECT * FROM permissions` and dynamically groups the matrix by the `module` string column using Java Streams.

### 9. Audit Service
- **Activity Log & JSONB Storage:** Created an `AuditLog` entity mapping exactly to the `audit_logs` table schema. Implemented Hibernate 6's `@JdbcTypeCode(SqlTypes.JSON)` to properly serialize `old_value` and `new_value` Maps into queryable `JSONB` columns in PostgreSQL.

### 10. File Management Service
- **Schema Alignment:** Renamed the outdated `FileMetadata` entity to `File`, pointing it explicitly to the `files` table. Updated all 9 columns to perfectly mirror the new database schema (e.g. `file_size`, `mime_type`, `storage_path`, `uploaded_by`).
- **File Upload/Download API:** Refactored `FileService` and `FileController` to persist the new `File` records upon upload, explicitly capturing the authenticated `uploaded_by` User UUID directly from the Spring Security Context (`@AuthenticationPrincipal`).

### 11. Settings Management Service
- **New Entity & Mapping:** Introduced a new `Setting` entity strictly mapped to the new `settings` table schema mapping columns `setting_id`, `organization_id`, `setting_key`, `setting_value` (TEXT), and `category`.
- **API Construction:** Fully built out `SettingRepository`, `SettingService`, and `SettingController` to expose a complete suite of authenticated CRUD operations for managing dynamic organization configurations. integrated Audit Logging directly on Save and Delete operations.
### 12. Company Application Management
- **Module Subscriptions:** Designed the `CompanyApplication` bridge entity to establish a many-to-many relationship between `Company` and global `Application`s (e.g., Sales, Account Finance, Ticketing).
- **Backend Orchestration:** Extended the `Company` entity to lazily fetch its active module subscriptions and established the `CompanyApplicationRepository`.
- **API Extension:** Updated `ApplicationController` and `ApplicationRegistryService` to expose a `PUT /api/admin/applications/companies/{companyId}` endpoint for authorized Org Admins to securely toggle modular features for their subsidiary companies, fully tracked by the `AuditService`.

### 10. File Storage Service
- **Centralized Metadata:** Created a `FileMetadata` entity to securely track original filenames, content types, sizes, and their tenant organization.
- **Secure File Storage:** Built `FileService` which intercepts multipart uploads, assigns a collision-proof UUID, and saves the file to a secure local `uploads/` directory on the server.
- **Upload/Download APIs:** Built `FileController` exposing `POST /api/admin/files/upload` and `GET /api/admin/files/{fileId}`. The download API automatically forces correct Content-Disposition headers for seamless viewing/downloading in the browser.
- **Enterprise Limits:** Raised multipart configuration limits in `application.yaml` to 50MB to gracefully handle high-resolution company logos, stamps, and large documents.

### 11. Notification Service (Mocked)
- **Async Delivery:** Created `NotificationService` with `@Async` methods to ensure email dispatching never blocks the main API threads.
- **Mocked Implementation:** Temporarily mocked the `sendPasswordResetEmail()`, `sendWelcomeEmail()`, and `sendEmployeeInviteEmail()` methods to securely output to the SLF4J console log, allowing platform development to continue uninterrupted without requiring an active SMTP server.
- **Test Endpoints:** Built `NotificationController` with test endpoints so admins can manually trigger and verify the mock email flows.

### 12. Application Registry
- **Rich Metadata Entity:** Refactored the `Application` entity to match the full database schema (`application_code`, `application_name`, `application_route`, `icon`, `description`, `status`) and implemented an `ApplicationRequest` DTO with validation.
- **Employee Mapping:** Enhanced the `Employee` entity by adding a `@ManyToMany` relationship with `Application`, allowing the system to track precisely which employee can access which software module (HRMS, Finance, Sales, etc.).
- **Registry API:** Built `ApplicationController` and `ApplicationRegistryService` to expose endpoints for creating new system applications and dynamically assigning them to employees.
- **Dashboard Data:** Completely removed the last piece of mocked data from the platform! `DashboardService` now fetches the true count of registered applications directly from the database.

### 13. Enterprise RBAC
- **Organization Scoping:** Updated the `User` entity to belong to an `Organization` instead of a `Company`. Users now manage resources across the entire scope of their Organization.

### 14. True JPA Auditing & Soft Deletion
- **Automated Timestamps:** Enabled Spring Data JPA `@EnableJpaAuditing` to automatically populate `created_at` and `updated_at`.
- **Soft Delete:** Implemented `@SQLDelete` and `@SQLRestriction` on the core entities. Deleting them now safely populates the `deleted_at` column instead of destroying rows, providing a complete audit trail.

## 🛠️ Testing the Endpoints (cURL Commands)

### 1. Bootstrap the Super Admin
```bash
curl -X POST http://localhost:8080/api/auth/setup-super-admin \
  -H "Content-Type: application/json" \
  -d '{ 
        "name": "Super Admin", 
        "email": "admin@grivety.com", 
        "password": "SecurePassword123" 
      }'
```

### 2. Login as Super Admin
```bash
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{ "email": "admin@grivety.com", "password": "SecurePassword123" }'
```
*(Copy the `token` from the response. You will need it for the next requests)*

### 3. Create an Organization
```bash
curl -X POST http://localhost:8080/api/admin/organizations \
  -H "Authorization: Bearer YOUR_SUPER_ADMIN_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ 
        "organizationCode": "ORG001", 
        "organizationName": "Grivety Global",
        "legalName": "Grivety Global Pvt Ltd",
        "organizationType": "Pvt Ltd",
        "industry": "Technology",
        "status": "Active"
      }'
```
*(Copy the `id` of the organization from the response. You will need it for the next requests)*
